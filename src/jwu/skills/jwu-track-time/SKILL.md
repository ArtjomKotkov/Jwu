# jwu: трекинг времени в Jira (worklog)

## Overview

Залогировать в таймтрекер Jira (worklog) время по задачам за день. Список задач берётся
из итогов дня (как в `jwu-post-analyze-day`), **плюс ревью** чужих PR, которые пользователь
посмотрел (апрувнул / поставил needs work — описание «Ревью»), **плюс** любые дополнительные
задачи, которые пользователь продиктует с описанием. По каждой задаче спрашиваем потраченное
время, показываем **сводный план** и **ничего не трекаем до явного подтверждения**.

Логирование делается командой `jwu worklog <KEY> "<время>" -m "<описание>"` —
по одной задаче за раз, только после «да».

## Шаги

1. **Собери задачи дня.** Если итоги дня уже есть в контексте (после
   `jwu-post-analyze-day`) — переиспользуй их. Иначе собери сам тем же источником
   (локальная БД jwu, без сети); дату можно передать аргументом
   (`/jwu-track-time 2026-05-21`), иначе сегодня:

   ```bash
   today=${1:-$(date +%F)}
   jwu jobs --json | jq --arg d "$today" '
     map(select(any(.records[]?; .ts[0:10] == $d) or (.updated_at[0:10] == $d)))
     | map({task_key,
            done: [.records[]? | select(.ts[0:10] == $d) | "[\(.kind)] \(.text)"]})'
   ```

   Для каждой задачи сделай **сухое описание** того, что реально сделано по сути
   (это и пойдёт в комментарий worklog). Действуют те же правила, что в
   `jwu-post-analyze-day`: **без рутины** — не пиши про прогон ревьювера (нейропрогон),
   коммиты, пуши, переключение веток; тесты — только если это была заметная отдельная
   работа, а не штатный прогон после правок; конфликты/мердж — писать стоит.

2. **Собери ревью, которые пользователь посмотрел за день.** Это PR'ы, где статус
   пользователя — апрув (галка) или needs work, **с датой ревья за нужный день**. Дата
   ревья берётся из ленты активностей PR — её отдаёт сам jwu, отдельная команда:

   ```bash
   today=${1:-today}     # 'today' или конкретная дата YYYY-MM-DD из аргумента скилла
   jwu prs --mine-reviews --on "$today" --json | jq -r '
     .[] | "\(.title | capture("(?<k>[A-Z][A-Z0-9]+-[0-9]+)").k // "—")\t\(.my_review_status)\t\(.title)"'
   ```

   `--on today` (или `--on 2026-05-21`) оставляет только ревью, поставленные в этот день
   (по дате моего апрува/needs work из activities, локальная дата). `--mine-reviews` сам
   фильтрует по моему статусу `APPROVED`/`NEEDS_WORK` и по логину из конфига — руками
   ничего сопоставлять не надо. Без `--on` команда отдаст все мои открытые ревью с датами.

   Ключ задачи берётся из заголовка PR (`ABC-123`). Один и тот же ключ из нескольких PR —
   объединяй в одну строку «Ревью».

   Покажи найденное компактным списком-кандидатом (`KEY — галка/needs work — короткий
   заголовок`) и спроси, сколько времени по каждому ревью (время пользователь укажет сам).
   Каждое ревью идёт в план как строка `KEY — Ревью — время` (описание ровно «Ревью»).
   Если за день ревью нет (пустой вывод) и пользователь ничего не продиктовал — **строк
   ревью не добавляй вообще**. Внимание: видны только OPEN PR — PR, который ты апрувнул и
   который сегодня уже замержили, в выдачу не попадёт; такие пользователь может
   продиктовать на шаге 3.

3. **Прими дополнительные задачи.** Спроси пользователя, есть ли что затрекать сверх
   найденного (встречи, исследование, разбор инцидента, ревью уже замердженного PR и т.п.).
   Каждая такая запись — это `КЛЮЧ-ЗАДАЧИ` + краткое описание. Worklog пишется только
   в реальную задачу Jira, поэтому ключ обязателен; если пользователь даёт описание
   без ключа — спроси, в какую задачу его трекать.

4. **Проверь, что уже затрекано за день.** По всем собранным ключам (задачи дня + ревью +
   доп.задачи) глянь существующие worklog'и за тот же день — чтобы не задвоить. Команда
   читает Jira и оставляет только **мои** записи за дату:

   ```bash
   jwu worklogs --on "$today" KEY1 KEY2 KEY3 … --json
   ```

   Вернёт `{KEY: [{time, comment, started}, …]}` только по задачам, где у меня за этот день
   уже есть worklog (пустой объект — ничего не залогировано). Каждую такую задачу **помечай
   в плане** («уже затрекано: 2h — …») и **по умолчанию не трекай повторно**: спроси
   пользователя, добавить ли время сверх или пропустить. Сам ничего не дублируй.

5. **Узнай время по каждой задаче.** Если пользователь уже назвал часы — используй их.
   Если нет — спроси, сколько времени потрачено на каждую. Время в формате Jira:
   `2h 30m`, `45m`, `1h`, `1d 4h` (1d = рабочий день). Не выдумывай часы сам.

6. **Покажи сводный план и спроси подтверждение.** Выведи нумерованный список —
   одна строка на задачу: `№ — KEY — описание — время`, и итоговую сумму. Строки ревью
   идут с описанием «Ревью». Уже затреканное за день показывай отдельной пометкой и в
   сумму к трекингу **не включай**. Затем задай вопрос **«Трекаем?»**. До этого момента
   **ни одной команды `jwu worklog` не запускай**.

   ```
   1. PROJ-12593 — перенёс фикс stat v2 в release-10.8 — 2h 30m
   2. PROJ-399   — разрешил конфликт, откатил low-memory конфиг — 1h 15m
   3. PROJ-450   — разбор инцидента с очередью Kafka — 45m
   4. PROJ-471   — Ревью — 30m
   Итого к трекингу: 5h 00m

   Уже затрекано сегодня (повторно не трекаю):
   • PROJ-588 — 2h — Ревью

   Трекаем?
   ```

   Если пользователь правит состав/время/описания — обнови план и спроси «Трекаем?»
   заново. Любой ответ, кроме явного согласия, — не трекаем.

7. **После подтверждения — логируй.** Для каждой строки выполни:

   ```bash
   jwu worklog PROJ-12593 "2h 30m" -m "перенёс фикс stat v2 в release-10.8"
   ```

   Команда сама ходит в Jira (worklog REST). Идёт по задачам по очереди; если какая-то
   упала (неверный ключ/формат/нет прав) — продолжай остальные и в конце отчитайся:
   что затрекано, что нет и почему. Сумму затреканного покажи итогом.

## Заметки

- Пишем в Jira (worklog) — это **не read-only**, поэтому строгое подтверждение перед
  запуском обязательно. До «да» — только чтение и показ плана.
- Формат времени — Jira-нотация в одну строку: `"1d 4h 30m"`. Часы/минуты, а не доли.
- Это про **залогировать** время. Просто текстовая выжимка дня без записи — это
  `jwu-post-analyze-day`. Что делать дальше / на ревью — `jwu-analyze-day`.
- `jwu worklog --help` — параметры: `-m/--comment` описание, `--started` ISO-время
  начала (по умолчанию — текущий момент), `--json` для машинного вывода.
- Ревью трекаются в **ту задачу, чей PR смотрели** (ключ из заголовка PR), с описанием
  ровно «Ревью». Источник — `jwu prs --mine-reviews --on <дата>` (только OPEN PR): команда
  фильтрует по моему статусу `APPROVED`/`NEEDS_WORK` и по дате моего ревья (из activities).
  Поля в JSON: `my_review_status` и `my_review_at` (ms). Время по каждому ревью указывает
  пользователь — сам часы не выдумывай.
- Защита от двойного трека: `jwu worklogs --on <дата> KEY…` показывает мои уже залогированные
  worklog'и за день по переданным ключам (read-only). Что там есть — повторно не трекаем,
  если пользователь явно не попросил добавить сверх.
