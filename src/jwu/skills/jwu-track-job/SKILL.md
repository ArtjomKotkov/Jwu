# jwu: ведение работы (job) — записи прогресса и результатов

> **Источник истины по типам — код.** Список валидных `--kind` и их бейджи живут в
> `src/jwu/core/models.py` (`JOB_RECORD_KINDS`, `JOB_RECORD_BADGES`) репозитория
> jwu. Добавляешь/переименовываешь тип — правь **и код, и этот скилл** (таблицу ниже
> + триггеры). Рассинхрон = `jwu job add --kind <новое>` упадёт на `click.Choice`.

## Когда

Во время активной работы над задачей, когда есть что зафиксировать в текущей **работе (job)**:
отметить фазу, записать найденный баг или его фикс, прогон тестов, запрет/риск, договорённость.
Также — по прямым фразам пользователя (см. «Триггеры»). Работу создаёт **jwu-start-job**; здесь — её ведение.

## Перед записью

- Нужен `JOB_ID` текущего цикла. Не знаешь — `jwu jobs --task <KEY>` (бери активную) либо спроси.
  **Не плоди новые работы ради заметки** — новая работа = новый цикл правок, это делает jwu-start-job.
- Контекст потерян / новая сессия / вернулись к задаче — сначала подхвати состояние работы
  через **jwu-resume-job**, потом логируй.
- Пиши СУТЬ, не дамп: что / где / итог. Одна мысль — одна запись.
- Запись фиксируется командой `jwu job add <JOB_ID> --kind <тип> "<текст>"` (+ опц. `--status`).

## Типы записей

| тип | бейдж | когда использовать |
|---|---|---|
| `phase` | — | веха процесса: Ф1 мердж → Ф2 анализ → Ф3 правки → Ф4 коммит. Статус: `--status pending\|done`. |
| `note` | · | нейтральный контекст/факт/прогресс (стенд, доступы, наблюдения). Дефолт. |
| `decision` | 🧭 РЕШЕНИЕ | **наш выбор подхода с обоснованием** («решили делать так, потому что…»): выбранный вариант, причина, отвергнутые альтернативы. |
| `remark` | ✎ | замечание **ревьюера** из PR (а не свой вывод!). |
| `constraint` | ⛔ ЗАПРЕТ | чего делать **НЕЛЬЗЯ**: «не трогать X», «гонять без Y», «не пушить в Z». |
| `warning` | ⚠ ВНИМАНИЕ | риск/потенциальная проблема — ещё не подтверждённый баг и не запрет. |
| `bug` | 🐛 БАГ | **найденный** баг: симптом, где, как воспроизвели. |
| `bug-resolved` | ✅ БАГ ИСПРАВЛЕН | баг **починили** — отдельной записью с сутью фикса. |
| `test-pass` | 🧪 ТЕСТЫ OK | прогон тестов **прошёл**: что гоняли + итог. |
| `test-fail` | 🧪 ТЕСТЫ УПАЛИ | прогон **упал**: команда, сколько/какие упали, суть ошибки. |
| `todo` | 📌 TODO | отложенная **актионабельная** работа в рамках цикла: что ещё доделать. |

Бейджевые типы подсвечиваются и в `jwu job show`, и в дашборде (вкладка «Работы»).

## Дисциплина ведения

- **Фазы.** Фиксируй каждую веху `--kind phase`, закрывай `--status done`. Большую задачу веди по одной фазе (Ф1 мердж/конфликт — всегда первой).
- **Баги «нашли → исправили».** На находку — `bug`; на фикс — **парный** `bug-resolved` ОТДЕЛЬНОЙ записью. Исходную `bug`-запись НЕ правь и НЕ удаляй — нужна история «нашли → исправили».
- **Тесты.** На каждый значимый прогон — отдельная запись по результату (`test-pass` / `test-fail`). Если падение = дефект кода, заведи ещё и парный `bug`.
- **Запрет vs предупреждение.** Жёсткое «нельзя» → `constraint`; «осторожно, может сломаться / под вопросом» → `warning`.
- **Решение vs запрет.** `decision` — это НАШ выбор с обоснованием («решили чинить парсер, а не править конфиг на стенде, потому что…»); `constraint` — внешнее «нельзя». Если поймал себя на том, что пишешь «почему сделали так» в `note` — это `decision`.
- **`todo` — только in-cycle.** Отложенное в рамках текущего цикла. Работа закрывается `done`, поэтому «на следующий цикл / на потом вообще» пиши не сюда, а `jwu note <KEY>` на самой задаче — иначе потеряется. Закрытый пункт можно пометить `--status done`.
- **Не подменяй тип на `note`/`remark`,** если по сути это решение / баг / тест / запрет / предупреждение / todo. Тип несёт смысл — по нему потом фильтруют и читают лог.
- **`remark` только для ревью.** Свои находки/блокеры — это `bug`/`warning`, а не `remark`.

## Триггеры по фразе пользователя (тип определяется фразой)

- «запиши решение / зафиксируй, почему … / решили …» → `--kind decision`
- «добавь ограничение / констрейнт / зафиксируй запрет» → `--kind constraint`
- «добавь todo / надо ещё … / не забыть …» → `--kind todo`
- «нашли баг / запиши баг / есть баг …» → `--kind bug`
- «баг исправлен / починили баг / резолвнут» → `--kind bug-resolved`
- «добавь предупреждение / warning / осторожно с …» → `--kind warning`
- «запиши прогон тестов / тесты прошли / тесты зелёные» → `--kind test-pass`; «тесты упали / красные» → `--kind test-fail`
- «отметь фазу … (done)» → `--kind phase` (+ `--status done`)

## Прочие команды работы

- `jwu job link <JOB_ID> --pr <id> --project <KEY> --repo <slug>` — привязать PR к работе.
- `jwu job status <JOB_ID> <active|paused|done|cancelled>` — сменить статус; `jwu job done <JOB_ID>` — завершить цикл.
- `jwu job show <JOB_ID>` — посмотреть лог по времени.

## Подтверждение правок

Ведение записей — локальная память, подтверждения не требует. Но **правки КОДА, коммиты и пуш — только после явного подтверждения пользователя** (правило проекта). Подробности старта/анализа — в скилле **jwu-start-job**.
