# jwu: разбор рабочего дня

## Overview

Краткий разбор рабочего дня по данным из `jwu` (Jira + Bitbucket). Утилита делает фулл-синк и
отдаёт расширенный контекст + промпт; ты составляешь **сжатую сводку-план** и сохраняешь её.
НЕ углубляйся в детали — пользователь хочет обзор «что случилось и что делать», подробности он разберёт сам.

## Шаги

1. Выполни команду и прочитай её stdout целиком (это контекст + инструкция):

   ```bash
   jwu action day-analyze
   ```

   Команда сама делает фулл-синк (может занять ~30-90с). Если она падает с ошибкой авторизации —
   сообщи пользователю и остановись (нужны токены/доступ, см. `jwu auth check`).

2. Проанализируй данные, **сочетая два среза**:
   - **текущие дельты** (раздел «Изменения с прошлого синка») — что произошло нового;
   - **текущее состояние** каждого PR и задачи — даже без свежей дельты состояние может
     требовать действия (например, PR давно апрувнут, но задача всё ещё не на тестах).

   Поля контекста, на которые опираются эвристики: в строке PR — `состояние:`
   (`конфликт` / `есть NEEDS_WORK` / `нет ревьюверов` / `ждёт апрувов` / `апрувы собраны`),
   `ревью:` (`имя:A` = апрув, `:NEEDS_WORK`, `:N`/прочее), `комментов:`, `обновлён:`
   (как давно); у задач — `[статус]`, `assignee:`; в «Упоминаниях» — пометка
   `· ждёт ответа` (последний коммент не мой). Шапка содержит моё имя — по нему отличай
   «на мне» от «не на мне».

   Для каждого «моего» PR и задачи выведи **конкретный следующий шаг**, опираясь на
   состояние, а не только на дельту. Применяй эвристики (состояние → что написать):

   PR:
   - **Конфликт** (`состояние: конфликт`, или дельта `new_conflict`) → «поправить
     merge-конфликт». Если при этом апрувы собраны — это приоритет (иначе апрувы впустую).
   - **`апрувы собраны`, нет `NEEDS_WORK`, нет конфликта, НО задача не в статусе тестирования**
     → «PR готов — перевести задачу на тесты» (сменить статус). Статус задачи бери из «Мои
     задачи» по ключу из заголовка PR; «тестовые» статусы — содержащие TEST/ТЕСТ или вида
     «READY FOR TESTING» (названия зависят от проекта).
   - **`апрувы собраны`, конфликта/`NEEDS_WORK` нет, задача УЖЕ на тестах** → «готово,
     действий не нужно» (отметь явно, чтобы не висело).
   - **`есть NEEDS_WORK` / новые замечания** (дельта `new_pr_comment`) →
     «ответить/поправить по замечаниям»; коротко суть из комментов PR.
   - **Смешанное ревью** (есть и `:A`, и `:NEEDS_WORK`) → «закрыть замечания, не мержить».
   - **`нет ревьюверов`** → «назначить ревьюверов / попросить ревью».
   - **`ждёт апрувов` и много комментов, но апрувов нет** → ревью буксует, пнуть/уточнить.
   - **Новые коммиты после апрува** (дельта `new_pr_commit` при уже собранных апрувах) →
     «нужен переревью».
   - **Stale-PR** (`обновлён:` давно — недели) → «разморозить или закрыть».

   Ревью (раздел «PR на ревью» — там ты ревьювер):
   - **Ждут моего ревью** (нет ни одного апрува/ревью) → глянуть, это на мне; приоритет,
     если висит давно (`обновлён:`).

   Задачи и упоминания:
   - **Упоминание (не на мне), особенно на тестах, где меня тегнули с просьбой/ожиданием**
     → уточни, ЧТО именно от меня хотят (из текста упоминания), предложи ответ/действие.
     Пометка `· ждёт ответа` усиливает приоритет. Не смешивай с «моими» задачами.
   - **Задача на тестах и на мне** (`assignee:` = я) висит давно → «пнуть тестирование /
     проверить статус».
   - **`status_change`, сделанный не мной** → проверь, не требует ли это теперь действия от
     меня (например, вернули из тестов → доработать).
   - **Задача `resolved`/закрыта** (дельта `resolved`) → «закрыть работу, убрать из активных».
   - **Приоритетная задача** (High/Highest) без движения → вынеси наверх плана.

   База — события из дельт: новые комменты (`new_comment`/`new_pr_comment`), смена статуса
   (`status_change`), новые апрувы (`reviewer_approved`), новые PR (`new_pr`).

   Затем собери **краткий план** маркерами, сгруппировав по действиям (пустые разделы опускай):
   - **Срочно поправить** — конфликты, `NEEDS_WORK`.
   - **Двигать дальше** — апрувнутые PR → перевести задачу на тесты; задачи, ждущие смены статуса.
   - **Готово / можно мержить** — PR без блокеров (кратко, чтобы закрыть в голове).
   - **Ответить / упоминания** — что от меня просят и где (особенно `· ждёт ответа`).
   - **Глянуть на ревью** — PR, ждущие моего ревью.
   - **Прочие изменения** — остальное из дельт.

   Пиши сжато, без воды и без копирования всего контекста. Где уместно — указывай ключ
   задачи / `repo#id` PR. Не выдумывай: если для эвристики не хватает данных (например,
   не виден статус задачи по PR) — не утверждай, а отметь как «проверить».

3. Сохрани план в историю (передай текст плана в stdin):

   ```bash
   jwu analysis save --title "День $(date +%Y-%m-%d)" <<'EOF'
   <твой план>
   EOF
   ```

4. Покажи план пользователю и скажи, что он сохранён (его можно открыть во вкладке «Анализ»
   в `jwu dashboard` или командой `jwu analysis show --last`).

## Заметки

- Это read-only по отношению к Jira/Bitbucket: только чтение + сохранение плана в локальную БД `jwu`.
- Прошлые планы: `jwu analysis list` / `jwu analysis show <ID>`.
- Если данных мало (пустые разделы) — так и напиши коротко, не выдумывай.
